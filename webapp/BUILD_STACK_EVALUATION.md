# Web 构建栈升级评估（P1）

## 1. 当前状态与风险

- 当前前端构建栈：`webpack@4` + `webpack-dev-server@3`（见 `webapp/app/package.json`）。
- 风险点：
  - 生态老旧，长期维护成本与安全修复成本持续上升。
  - wasm + 静态资源同步依赖脚本拼装，构建链可读性一般。
  - E2E 与发布流程的集成尚处于最小可用状态，后续扩展时复杂度会增加。

## 2. 升级候选方案

### A. 方案一：先升级 webpack 生态（低风险）
- 目标：升级到较新 webpack 生态，尽量保持目录与运行方式不变。
- 优点：改动可控，迁移成本低。
- 缺点：仍需维护 webpack 配置与兼容层，长期收益有限。

### B. 方案二：迁移到 Vite（推荐中期目标）
- 目标：开发体验、构建速度、配置复杂度全面优化。
- 优点：更现代、更轻量，社区对 wasm 场景支持成熟。
- 缺点：需要改造 dev server、静态资源策略与构建脚本，短期改动较大。

## 3. 建议路线图（分阶段）

1. 阶段 0（已完成）：建立前端单一事实源 + 最小 E2E + CI 门禁。  
2. 阶段 1：在不迁移工具链的前提下，升级依赖到可稳定版本并清理告警。  
3. 阶段 2：建立 Vite PoC 分支（保持 UI/功能一致，验证 wasm 打包与静态资源输出）。  
4. 阶段 3：对比指标后决定是否切换默认构建链。  

## 4. 回滚方案

- 保留 webpack 构建脚本和 CI job，Vite 迁移期间并行验证。
- 若关键指标不达标，直接回退到 webpack 主链路，不影响 `docs` 发布产物目录结构。

## 5. Go / No-Go 决策门槛

- 构建时间：本地与 CI 构建时间较当前方案至少不恶化（推荐提升 ≥20%）。
- 兼容性：wasm 产物加载、样例导入、SVG/PDF 导出、E2E 冒烟全部通过。
- 维护成本：配置文件数量与脚本复杂度明显下降。
- 安全性：依赖漏洞数量下降，且无高危未修复项。
